## Synthèse

Les constats se regroupent en quelques causes racines récurrentes. Traiter la cause
plutôt que chaque symptôme isolé est le levier le plus rentable.

### 1. Deux suppressions de surface langage non propagées (SPEC + commentaires)

Deux features retirées du langage continuent d'être documentées comme vivantes :

- **Suffixes numériques typés** (`42i32`, `0xFF_u8`). Le lexer les *rejette* désormais
  (`MalformedNumericLiteral`), et `SPEC.md:233/241` confirme « there is no typed-suffix
  form ». Pourtant : la vaderdoc de `Token` décrit un champ `numeric_suffix` inexistant ;
  l'en-tête de `lexer.vader` annonce un « optional suffix » ; le commentaire de
  `emit_int_lit`/`emit_float_lit` parle de slices « incluant le suffixe » ; et
  `std/numbers::is_int_suffix`/`is_float_suffix` (toujours dans la SPEC) n'existent plus.
- **Renommage `as_string`→`bytes_to_string`** (l'inverse de `bytes()`). La SPEC le nomme
  toujours `as_string` à trois endroits (`§587`, `§2396`, `§1966`) — y compris dans un
  exemple d'import — alors qu'aucun `as_string` libre n'existe (le seul est une méthode
  sur `Path`). Du code copié depuis la SPEC ne compile pas. **C'est la dérive de SPEC la
  plus grave.**

### 2. Le `src/` TypeScript supprimé hante les commentaires et la SPEC

Le compilateur TS sous `src/` a été supprimé (2026-06-22), mais il reste cité partout
comme « source de vérité » à mirrorer, et la SPEC se contredit sur l'état du self-host.
`SPEC.md:66` dit que le typer/lower/mid-IR/C-emit « not yet ported, src/ remains the
reference » tandis que `SPEC.md:2955` dit « the full native pipeline is self-hosted ».
Les fichiers lexer/diagnostics/resolver/c_emit/midir/cli affirment des invariants de
parité (`Mirrors src/…`, `byte-identical with the TS reference`, `tests/parity.test.ts`)
contre un arbre et un harnais de tests qui n'existent plus. Un seul nettoyage (purger
les références `src/` + aligner les deux phrases d'état) résout une douzaine de constats.

### 3. La règle « comparer les enums avec `==` natif » violée massivement

`CLAUDE.md §3` est une règle dure : `x == Variant`, jamais `i32(x) == i32(Variant)`.
Le balayage trouve ~53 sites en forme castée contre ~116 en forme native — souvent dans
le même fichier ou le même module qu'un voisin conforme. Concentrations : `SymbolKind`
(resolver `collect`/`prescan`/`wire`, comptime `eval_ident`/`check`/`deps`), `BinaryOp`/
`LoweredBinaryOp`/`IterChainKind` (typecheck, midir `build`, lower `lower_for_in`). Le
justificatif d'origine (« SymbolKind est gardé en i32 plutôt qu'en enum ») est lui-même
faux : `SymbolKind :: enum(i32)`. Un sweep unique sur l'arbre éteint toute la famille.

### 4. Un concept, deux fonctions parallèles qui ont divergé

Plusieurs calculs existent en double exemplaire qui ne sont plus d'accord :

- **`@extern` nom de symbole C** : le typer lit le *dernier* arg dès qu'il y en a un
  (forme 1-arg honorée), le générateur midir exige `< 2 → bail` (forme 1-arg ignorée).
  → mauvais symbole C / échec de lien (HAUTE).
- **`@type_kind`** : `type_kind_label` (comptime) vs `kind_string_of_type` (lower)
  divergent sur `TypeParam`/`Any`/`TypeMeta` ; la table comptime invente `"type_param"`
  (hors SPEC) et n'émet jamais `"type"`/`"any"`.
- **Narrowing de chaîne de champs** : `stmt.vader` compose contre le type narrowé courant
  (`narrow_for_field`), `expr_if.vader` soustrait du type déclaré brut.
- **Précédence `is`/`in`/`!in`** : table du printer (rang 4, plus serré que les
  comparaisons rang 3) contre Pratt du parser (tous à `left_bp=20`, non-assoc).
- **Résolution d'import** : `lazy`/`collect` (modèle include-path, rejette `./`) contre
  `resolve_import_path` (ancien modèle relatif + préfixe stdlib).
- **Tables `c_type_for_array_kind` (morte) vs `typed_array_fill_for` (vive)**, prédicats
  d'emptiness, marche AST hover vs code-actions, etc.

### 5. Les commentaires de « stade phase 1 / pas encore porté » survivent au portage

Une famille entière de constats : des en-têtes/vaderdoc décrivent du travail comme futur
ou non câblé alors que le code complet est juste en dessous — `materialize`/`substitute`
(clonage de corps « not yet ported »), `cfg_dce` (`Future passes`), `emit_instr`
(`minimal subset`), `bc_type_of` (`Phase 2`), `lower_for_in` (fast paths « skipped »),
`emit_fn_body`/`emit_project` (référencés mais inexistants), `LoadF64` (cite une fonction
supprimée). Aucun impact runtime, mais ils trompent activement le lecteur.

### 6. Surface langage documentée mais non implémentée

La SPEC promet plus que le code ne livre : **arguments nommés** `f(a=1, b=2)` (le typer
est positionnel, ignore `CallArg.name`) et **paramètres par défaut** `greet("Alice")`
(arité exacte exigée, `FnType` n'a pas de min-arité) — tous deux HAUTS, avec exemples
SPEC qui échouent. Plus : `@field_count`/`@variant_count`/`@field_index` (enregistrés,
jamais foldés), `--target=ir-text` (annoncé, rejeté), options globales `--allow-env`/
`--no-bytecode-opt`/`--diagnostics` (parsées, jamais consommées), `R2005 ImportCycle`
(déclaré, jamais émis — aucune détection de cycle n'existe).

### 7. Bugs de round-trip du formateur (paren-drop)

Le printer ne reparenthése pas plusieurs constructions, produisant une sortie qui ne se
reparse pas à l'identique (ou plus du tout) : précédence `is`/`in` inversée, opérateurs
non-assoc traités comme assoc-gauche, `emit_range` sans garde de précédence, littéral de
struct en tête de condition `if`/`match`/`for`. La garantie « parse-equivalent » de
`format_source` est violée.

## Quick wins

Petits efforts, impact réel sur la correction ou la cohérence, par ordre de rentabilité.

1. **Corriger le seuil `@extern` 1-arg dans `midir/build.vader:198`** (`< 2` → lire le
   dernier arg dès `>= 1`, comme `decl.vader`). Une ligne ; supprime un mauvais symbole C
   silencieux et aligne typer/codegen/SPEC. (HAUTE)
2. **Renommer `as_string`→`bytes_to_string` dans la SPEC** (`§587`, `§2396`, `§1966`) et
   retirer `bytes`/`as_string` de l'exemple `import "std/string"`. Débloque le copier-coller
   depuis la doc. (HAUTE)
3. **Aligner les deux phrases d'état self-host de la SPEC** (`§66` vs `§2955`) et purger
   les références `src/` mortes (lexer, diagnostics, resolver, midir, c_emit, cli). Éteint
   ~12 constats de dérive documentaire d'un coup.
4. **Sweep `i32(x)==i32(Variant)` → `x == Variant`** sur les ~53 sites enum-cast (gardez
   les vrais unwrap de distinct comme `cfg.vader:37`). Règle dure de `CLAUDE.md §3`.
5. **Aligner les deux tables `@type_kind`** (`type_kind_label` ↔ `kind_string_of_type`) sur
   le jeu de chaînes stable de `SPEC.md:2293` (factoriser en une seule fonction).
6. **Composer le narrowing dans `expr_if.vader`** (chaîne de champs) via `narrow_for_field`
   comme `stmt.vader`, pour que les gardes `if x.f is A {…} else if …` se composent.
7. **Ajouter `ir-text` au garde de `cmd_build` (`main.vader:619`)** pour un message
   « not yet implemented » cohérent au lieu de « unknown target ».
8. **Supprimer le helper mort `c_type_for_array_kind`** (`c_emit/body.vader:454`) qui a
   dérivé de la table vive `typed_array_fill_for`.

## Incohérences par sous-système

### Lexer / surface lexicale

- **Référence `@extern` mise à part — voir Typecheck.**
- **Documentation des suffixes numériques typés supprimés** (medium) — `vader/lexer/token.vader:72-79`, `vader/lexer/lexer.vader:6-7,346-357`, `SPEC.md:233,241`. La vaderdoc de `Token` affirme qu'un champ `numeric_suffix` « coexists with value on numeric literals » alors que le struct n'a que `kind/text/span/value` (grep : `numeric_suffix` n'apparaît nulle part ailleurs) ; l'en-tête `lexer.vader` annonce « decimal int literals … with optional suffix » / « float literals … with optional suffix ». Or le lexer *rejette* désormais tout suffixe (`MalformedNumericLiteral`, « typed numeric suffix … is no longer supported ») et `SPEC.md` interdit la forme. *Correction :* supprimer la mention `numeric_suffix` et reformuler l'en-tête pour refléter le rejet des suffixes.
- **`==`/`!=` contre `null` ne déclenche pas T3040 — voir Typecheck.**
- **Underscore final `42_` accepté en silence** (medium) — `SPEC.md:234`, `vader/lexer/lexer.vader:267-298`. `SPEC.md` dit que `42_` est une erreur ; dans `consume_digits`, un `_` final suivi de EOF/espace/opérateur ne déclenche ni le garde de suffixe (ligne 279) ni le check `first || last_was_underscore` (282), donc aucun diagnostic ; `strip_numeric_text` retire ensuite le `_` et la valeur parse. Les `_` doubles et en tête sont, eux, attrapés. *Correction :* ajouter un check post-boucle sur `last_was_underscore` pour émettre l'erreur sur le `_` final.
- **Jeu d'échappement unique pour char et string** (low) — `vader/lexer/lexer.vader:389-409,471-472,644-645`, `SPEC.md:257,273`. Un seul `lex_escape` accepte inconditionnellement l'union `\n \t \r \\ \' \" \$ \0 \u{..}` ; la SPEC définit deux jeux distincts (char : pas de `\"` ni `\$` ; string : pas de `\'`). Le lexer accepte donc `'\"'`/`'\$'` et `"\'"`. *Correction :* paramétrer `lex_escape` par le type de littéral (char vs string) et rejeter les échappements hors-jeu (L0004).
- **En-tête « Deferred » contredit par le code** (low) — `vader/lexer/lexer.vader:12-14,456,531,572,615-641`. L'en-tête liste char literals, strings, raw/triple, interpolation comme « Deferred (will land in subsequent passes) » alors que `lex_char`/`lex_escape`/`lex_string`/`lex_string_body` + pile d'interpolation les implémentent tous. *Correction :* retirer le bloc « Deferred » devenu faux.
- **Token `$` (Dollar) non documenté en §3** (low) — `vader/lexer/lexer.vader:718`, `vader/lexer/token.vader:60`, `vader/parser/parser.vader:1099-1119`, `SPEC.md:294-313`. Le lexer émet un `.Dollar` autonome et le parser s'en sert pour la tête de paramètre-type comptime `($T, $N: i32)`, mais `SPEC.md §3` ne mentionne `$` que dans l'interpolation, et `SPEC.md:1065` interdit explicitement la forme paren. *Correction :* documenter le token `$` en §3 (ou faire converger la surface des params comptime).
- **Références `src/lexer/*.ts` mortes** (low) — `vader/lexer/token.vader:3`, `vader/lexer/keywords.vader:3`, `vader/lexer/dump.vader:32,133`. Affirment un mirror byte-identique d'un arbre `src/` supprimé. *Correction :* purger les mentions `src/` et l'invariant de parité TS.

### Parser

- **Lambdas valeur `fn(...) -> R { … }` acceptées** (medium) — `SPEC.md:1472,1495`, `vader/parser/parser.vader:2522,3129`, `vader/typecheck/expr_lambda.vader:33`. `SPEC.md §6` dit que le mot-clé `fn` n'est pas utilisé sur les lambdas et qu'il n'y a pas de slot de type de retour ; le parser route `.KwFn` vers `parse_lambda` qui lit un `-> return_type`, et le typer le consomme via `pick_return`. Forme dormante mais acceptée. *Correction :* rejeter la forme `fn`-lambda en position valeur (ou l'ajouter à la SPEC).
- **`CallArg.spread` jamais produit** (low) — `vader/parser/parser.vader:2692-2723`, `vader/parser/ast.vader:473-479`, `vader/parser/dump.vader:1056`, `vader/fmt/printer.vader:1241`, `vader/lsp/actions/ufcs.vader:53`. `parse_call_args` met toujours `.spread = false` et ne parse aucun `...` ; pourtant le champ est dumpé, formaté (`if a.spread { write("...") }`) et consulté par `ufcs`. Branches mortes côté consommateur. *Correction :* supprimer le champ `spread` de `CallArg` ou implémenter le spread d'argument.

### Resolver

- **Deux résolveurs d'import contradictoires** (medium) — `vader/resolver/module.vader:80-100`, `vader/resolver/collect.vader:228-243`, `vader/resolver/lazy.vader:54-104`, `vader/typecheck/orchestrate.vader:362-364`. Le loader vivant (include-paths) rejette `./`/`../` (R2025) et ne traite pas `std/` à part, mais `resolve_import_path` (vivant, appelé par `orchestrate`) implémente l'ancien modèle : préfixe `STDLIB_PREFIX` + import relatif. La même chaîne d'import est résolue de deux façons opposées. Latent aujourd'hui (le seul appelant saute les `std/` et le loader ne suit pas les relatifs). *Correction :* supprimer `resolve_import_path` et router le dump par le loader unifié.
- **Imports relatifs `./foo`/`../foo` contre la SPEC** (medium) — `vader/resolver/module.vader:1-15,87-99`, `SPEC.md §11`. L'en-tête et `resolve_import_path` documentent/implémentent une résolution relative + fallback filesystem que `SPEC.md §11` désavoue (« Relative paths … are not part of the language »), confirmé par le message R2025. *Correction :* supprimer le code/doc de feature désavouée (le garde R2025 en `collect` reste la voie correcte).
- **`R2005 ImportCycle` jamais émis** (medium) — `vader/resolver/loader.vader:44-47`, `vader/diagnostics/codes.vader:123,254`, `SPEC.md §11`. La vaderdoc de `LoadedProject` affirme « R2005 from cycle detection » et la SPEC dit « the resolver emits a diagnostic » pour les cycles, mais aucun site n'émet `.ImportCycle` ; la BFS de `drain_load_queue` dédoublonne les modules sans jamais détecter de cycle. *Correction :* implémenter la détection de cycle (pile in-progress) émettant R2005, ou retirer la promesse de la doc/SPEC.
- **`SymbolKind` comparé par cast malgré la doc « kept as i32 »** (medium) — `vader/resolver/symbol/symbol.vader:13-16`, `vader/resolver/collect.vader:124,206`, `vader/resolver/prescan.vader:62`, `vader/resolver/wire.vader:87`. - **Doc de clé `idents` fausse (`span_key`/i64 file-blind)** (medium) — `vader/resolver/body.vader:26-27,35-36`, `vader/resolver/resolve.vader:29-31`, `vader/diagnostics/diagnostic.vader:42`, `vader/resolver/scope.vader:130-146`. - **`materialize.vader` : clonage de corps « not yet ported » mais implémenté** (low) — `vader/resolver/materialize.vader:10-13,84-95,127`. L'en-tête dit que le clonage de corps de méthode par défaut n'est pas porté et que `body` reste null ; le code clone via `substitute_block_materialized` et assigne `.body` (non-null), câblé à `orchestrate.vader:764`. *Correction :* mettre l'en-tête à jour.
- **`substitute.vader`/`substitute_body.vader` : « runs on the TS side »** (low) — `vader/resolver/substitute.vader:13-16`, `vader/resolver/substitute_body.vader:16-18`, `vader/typecheck/orchestrate.vader:764`. Affirment que le body-walker n'est pas porté car « the Vader typechecker still runs on the TS side » ; il est porté, câblé, et le TS est supprimé. *Correction :* purger ces commentaires.
- **`dump.vader` : en-tête `## <displayPath>` mais code émet `m.path`** (low) — `vader/resolver/dump.vader:8-9,28`, `vader/resolver/resolve.vader:36-39`. L'en-tête promet le nom de module déclaré ; le code dumpe le chemin filesystem (comportement validé par snapshot). *Correction :* corriger l'en-tête en `## <path>`.
- **Commentaire `add_fn_symbol` inexistant** (low) — `vader/resolver/loader.vader:357-360`, `vader/resolver/collect.vader:101`. `merge_collected` réfère à la politique d'`add_fn_symbol` ; la fonction réelle est `add_fn`. *Correction :* corriger le nom.

### Typecheck

- **Arguments nommés `f(a=1, b=2)` non honorés** (high) — `SPEC.md:1454,1456`, `vader/typecheck/call.vader:20,236,287`, `vader/parser/ast.vader:476`. `SPEC.md §6` dit que les args nommés sont « always allowed » (l'exemple de params par défaut en dépend) ; le typer matche par index positionnel et ne lit jamais `CallArg.name` (l'en-tête l'admet : « positional only »). Un appel réordonné par nom se lie au mauvais slot. *Correction :* implémenter le matching par nom + l'arité min, ou restreindre la SPEC.
- **Paramètres par défaut non implémentés** (high) — `SPEC.md:1442,1449`, `vader/typecheck/call.vader:209`, `vader/types/types.vader:60`. `SPEC.md §6` montre `greet("Alice")` remplissant le défaut ; le typer exige l'arité exacte (T3003) et `FnType` ne porte ni défaut ni min-arité. Reproduit au binaire : l'exemple SPEC donne T3003. *Correction :* porter le défaut/optionnel dans `FnType` et le remplissage au site d'appel.
- **`@extern` 1-arg : typer honore, midir ignore** (high) — `vader/typecheck/decl.vader:363-375`, `vader/midir/build.vader:198-210`, `SPEC.md:2153,2181`. `extern_symbol_name_for` lit le dernier arg dès `len>=1` (résout `@extern("strlen") c_strlen` → `strlen`) ; `extern_symbol_name` fait `if dec.args.len() < 2 { continue }` et retombe sur le nom Vader (`c_strlen`). Le check de doublon T3050 se cale sur `strlen`, mais le C émis déclare `extern … c_strlen(…)` — mauvais symbole de lien, contraire à la SPEC. *Correction :* aligner midir sur la convention dernier-arg (`>= 1`).
- **`==`/`!=` contre `null` ne déclenche pas T3040** (medium) — `vader/typecheck/binary.vader:43-57,213-219`, `SPEC.md:1002`. `SPEC.md` dit que `==`/`!=` reçoivent le même check unreachable que `is` (`if n: i32 == null` → T3040) ; `check_is_overlap` n'est appelé que dans la branche `.Is`, jamais dans `check_equality` (dont la vaderdoc affirme même « s == null stays valid »). *Correction :* appeler `check_is_overlap` aussi dans le chemin égalité.
